1. Introduction to Ruby on Rails

What is Ruby on Rails?

Rails is a full-stack web framework that provides:

  • Convention over Configuration: Sensible defaults for rapid development
  • DRY Principle: "Don't Repeat Yourself" philosophy
  • MVC Architecture: Clean separation of concerns
  • Built-in Testing: Test-driven development support

üç∞ Cake Baking Analogy

Think of Rails as a professional kitchen where:

  • Chef = Rails Developer
  • Recipe Book = Rails Conventions
  • Ingredients = Gems (Libraries)

üìö Historical Context

Created by David Heinemeier Hansson in 2004, Rails has powered major platforms like:

  • GitHub
  • Shopify
  • Airbnb (initial version)

4. Routing Fundamentals

What are Routes?

Routes are URL patterns that map to controller actions. They work like:

üìÆ Post Office Analogy:

  • URL = Mailing address
  • Router = Postal sorting system
  • Controller = Recipient processing mail

  # config/routes.rb
  Rails.application.routes.draw do
    get '/about', to: 'pages#about'
    get '/contact', to: 'pages#contact'
  end

Code Explanation:

  • Rails.application.routes.draw do: This starts the routing definition block in Rails.
  • get '/about', to: 'pages#about': This defines a route for the URL /about. When a user visits this URL, the about action in the PagesController will be called.
  • get '/contact', to: 'pages#contact': Similarly, this defines a route for the URL /contact, which maps to the contact action in the PagesController.

RESTful Resource Routing

Standard CRUD routes for a resource:

HTTP Verb Path Controller#Action
GET /posts posts#index
GET /posts/new posts#new
POST /posts posts#create
GET /posts/:id posts#show
GET /posts/:id/edit posts#edit
PATCH/PUT /posts/:id posts#update
DELETE /posts/:id posts#destroy

üöÄ Quick Setup

Generate all REST routes with one line:


resources :posts  # Creates all 7 RESTful routes

Code Explanation:

  • resources :posts: This single line generates all seven RESTful routes for the posts resource. It automatically maps HTTP verbs to controller actions, as shown in the table above.
  • For example, GET /posts maps to posts#index, and DELETE /posts/:id maps to posts#destroy.

Advanced Routing Techniques

API Namespacing


namespace :api do
  namespace :v1 do
    resources :users
  end
end
# Creates /api/v1/users routes

Code Explanation:

  • namespace :api: This groups routes under the /api URL prefix.
  • namespace :v1: Further groups routes under /api/v1, which is useful for versioning APIs.
  • resources :users: Generates RESTful routes for the users resource under /api/v1/users.

Member Routes


resources :posts do
  member do
    get 'preview'
  end
end
# /posts/1/preview

Code Explanation:

  • member do: Defines a route that acts on a specific member (instance) of the resource.
  • get 'preview': Creates a route like /posts/1/preview, where 1 is the ID of the post. This route maps to the preview action in the PostsController.

Collection Routes


resources :posts do
  collection do
    get 'search'
  end
end
# /posts/search

Code Explanation:

  • collection do: Defines a route that acts on the entire collection of resources.
  • get 'search': Creates a route like /posts/search, which maps to the search action in the PostsController.

end

Runs after specified actions

around_action

Wraps around actions

7. Models & Active Record

What is Model?

üíæ Model:

    A Rails model is like a blueprint for a type of data in your app. It helps your app store, manage, and retrieve information from the database.

    For example, if you're building a blog, you might have a model called Post. This model would define what a post looks like and what it can do.

What is Active Record?

üíæ Active Record Pattern:

  • Maps database tables to Ruby classes
  • Each row = Object instance
  • Columns = Object attributes

# app/models/user.rb
class User < ApplicationRecord
  has_many :posts
  validates :email, presence: true, uniqueness: true
end

üß† Key Features:

  • Inheritance: All models inherit from ApplicationRecord
  • Associations: has_many defines relationships
  • Validations: validates ensures data quality

Database Migrations

üîç Explanation:

  • Migration Files: Versioned database schema changes
  • create_table: Generates new database table
  • t.timestamps: Adds created_at and updated_at columns
  • Run with: rails db:migrate

8. Model Associations

What are Associations?

üîó Definition:

Associations define relationships between models. They allow you to:

  • Connect database tables
  • Simplify data retrieval
  • Maintain data integrity

üìö Example Scenario

In a blog application:

  • A User has many Posts
  • Each Post belongs to a User
  • Posts can have many Comments

One-to-Many Relationship

1Ô∏è‚É£‚Üíüìö Definition:

A single record in one table can be associated with many records in another table. 


# app/models/user.rb
class User < ApplicationRecord
has_many :posts
end
                  

# app/models/post.rb
class Post < ApplicationRecord
belongs_to :user
end

üß† Key Points:

  • Foreign Key: posts table has user_id column
  • has_many: User can have many posts
  • belongs_to: Post is owned by one user

üíª Usage Example


# Create a user
user = User.create(name: "John")

# Create posts for the user
user.posts.create(title: "First Post")
user.posts.create(title: "Second Post")

# Retrieve user's posts
user.posts.each do |post|
puts post.title
end

Many-to-Many Relationship

üìö‚ÜîÔ∏èüìö Definition:

Records in one table can be associated with many records in another table, and vice versa. 


# app/models/post.rb
class Post < ApplicationRecord
  has_many :post_tags
  has_many :tags, through: :post_tags
end
                        

# app/models/tag.rb
class Tag < ApplicationRecord
  has_many :post_tags
  has_many :posts, through: :post_tags
end

üß† Key Points:

  • Join Table: post_tags connects posts and tags
  • through: Specifies the join model
  • has_many :through: Indirect association

üíª Usage Example


# Create tags
ruby = Tag.create(name: "Ruby")
rails = Tag.create(name: "Rails")

# Associate tags with a post
post = Post.create(title: "Rails Tips")
post.tags << ruby
post.tags << rails

# Find posts by tag
ruby.posts.each do |post|
  puts post.title
end

Polymorphic Associations

üîÑ Definition:

A model can belong to multiple other models on a single association. 


# app/models/comment.rb
class Comment < ApplicationRecord
belongs_to :commentable, polymorphic: true
end
                  

# app/models/post.rb
class Post < ApplicationRecord
has_many :comments, as: :commentable
end

# app/models/photo.rb
class Photo < ApplicationRecord
has_many :comments, as: :commentable
end

üß† Key Points:

  • commentable_type: Stores the associated model name
  • commentable_id: Stores the associated record ID
  • Flexible: Comments can belong to Posts or Photos

üíª Usage Example


# Create a post
post = Post.create(title: "New Post")

# Add comment to post
post.comments.create(body: "Great post!")

# Create a photo
photo = Photo.create(url: "example.jpg")

# Add comment to photo
photo.comments.create(body: "Nice photo!")

9. Active Record Query Interface

What is the Query Interface?

üîç Definition:

A set of methods to retrieve and manipulate data from your database without writing raw SQL. 


# Get all published posts
Post.where(published: true)

# Equivalent SQL:
# SELECT * FROM posts WHERE published = true

Essential Query Methods

where (Filtering)


Post.where("created_at > ?", 1.week.ago)

‚úÖ Safe from SQL injection

‚ùå Avoid: where("created_at > #{params[:date]}")

order (Sorting)


User.order(created_at: :desc)

Sort by multiple columns:

.order(:last_name, :first_name)

Limit & Pagination


Post.limit(5)  # First 5 posts
                      

# With pagination gem
Post.page(2).per(10)

Advanced Query Patterns

Scopes (Reusable Queries)


class Post < ApplicationRecord
  scope :published, -> { where(published: true) }
  scope :recent, -> { where("created_at > ?", 1.week.ago) }
end

# Usage:
Post.published.recent

Joins (Cross-Table Queries) 


Post.joins(:comments)
    .where(comments: { approved: true })

Equivalent SQL:

SELECT posts.* FROM posts INNER JOIN comments ON comments.post_id = posts.id WHERE comments.approved = true

Aggregations


Post.count  # Total posts
                            

Comment.group(:post_id).count
# {1 => 5, 2 => 3} (Post ID => Comment count)
                            

Order.average(:total_price)

Performance Considerations

Eager Loading


# Bad (N+1 Query)
Post.all.each { |p| p.comments.count }

# Good
Post.includes(:comments).each { |p| p.comments.size }

Pluck vs Select


# Returns array of emails
User.pluck(:email)  

# Returns User objects
User.select(:email)

üí° Database Indexes


class AddIndexToPosts < ActiveRecord::Migration[7.0]

  add_index :comments, [:post_id, :approved]
end
end

Indexes speed up queries but slow down writes

11. Database Transactions

What is a Database Transaction?

üîí Definition:

A transaction is a sequence of database operations that are treated as a single unit of work. Either all operations succeed, or none are applied.

üìö Real-World Analogy

Think of a bank transfer:

  • Debit from Account A
  • Credit to Account B
  • Both must succeed or fail together

Why Are Transactions Important?

üíæ Data Integrity

Ensures your database remains in a consistent state, even if errors occur. 


# Without transactions
def transfer_funds
account_a.debit(100)
account_b.credit(100) # Fails here
end

‚ùå Account A loses money, but B doesn't receive it

üö® Error Handling

Automatically rolls back changes if any operation fails. 


# With transactions
def transfer_funds
Account.transaction do
  account_a.debit(100)
  account_b.credit(100) # Fails here
end
end

‚úÖ Both debit and credit are rolled back

How to Use Transactions

Basic Transaction


def create_order(user, product)
  Order.transaction do
    order = user.orders.create!(status: 'pending')
    order.line_items.create!(product: product, quantity: 1)
    order.update!(status: 'completed')
  end
end

üß† Key Points:

  • create!: Raises exception on failure
  • transaction: Wraps all operations
  • Atomic: All or nothing

Nested Transactions


def complex_operation
  User.transaction do
    user = User.create!(name: "John")
    
    Account.transaction(requires_new: true) do
      user.accounts.create!(balance: 100)
      raise "Error" # Rollback only this block
    end
  end
end

üîç Behavior:

  • requires_new: Starts a new transaction
  • Outer transaction continues if inner fails
  • Useful for partial rollbacks

Advanced Transaction Patterns

Savepoints


def complex_operation
User.transaction do
  user = User.create!(name: "John")
  
  ActiveRecord::Base.transaction(requires_new: true) do
    user.accounts.create!(balance: 100)
    raise ActiveRecord::Rollback # Partial rollback
  end

  user.update!(email: "john@example.com")
end
end

üí° Use Cases:

  • Recover from partial failures
  • Continue outer transaction
  • Log errors without full rollback

Locking Records


def update_balance(account)
Account.transaction do
  account.lock! # Prevents concurrent updates
  account.update!(balance: account.balance + 100)
end
end

üîí Lock Types:

  • lock!: Exclusive lock
  • with_lock: Shorthand for lock + block
  • Optimistic Locking: Uses lock_version

13. Concerns & Mixins

Concerns in Rails

Concerns help organize reusable code that can be shared across multiple models/controllers.

üç∞ Real-World Analogy:

Think of concerns like a recipe book. Instead of writing the same recipe (code) in every kitchen (class), you keep it in one book and reference it when needed. 


# app/models/concerns/trashable.rb
module Trashable
  extend ActiveSupport::Concern

  included do
    scope :in_trash, -> { where(trashed: true) }
    scope :not_trashed, -> { where(trashed: false) }
  end

  def move_to_trash
    update(trashed: true)
  end
end

Full Explanation:

  • File Location: Stored in app/models/concerns or app/controllers/concerns
  • extend ActiveSupport::Concern: Gives Rails-specific features like included blocks
  • included do: Runs code when module is included in a class (sets up relationships/scopes)
  • Usage: Add include Trashable to any model needing trash functionality
  • Benefits: Avoid code duplication, easier maintenance

Ruby Mixins

Mixins add common functionality to classes using plain Ruby modules.

üîå Real-World Analogy:

Like USB ports on a laptop. You can plug in different devices (modules) to add functionality (keyboard, mouse) without rebuilding the laptop. 


module Exportable
  def to_csv
    CSV.generate do |csv|
      csv << column_names
      all.each do |item|
        csv << item.attributes.values
      end
    end
  end
end

class Product < ApplicationRecord
  include Exportable
end

# Usage:
Product.all.to_csv # Generates CSV data

Key Differences from Concerns:

  • Works in any Ruby class (not just Rails)
  • No special included blocks
  • Simpler code sharing without Rails magic

14. View Helpers

üßπ Why Use Helpers?

Helpers keep your views clean by moving complex logic out of HTML templates. Think of them as specialized tools in a toolbox - each has a specific job. 


# app/helpers/posts_helper.rb
module PostsHelper
  def reading_time(post)
    words_per_minute = 200
    words = post.content.split.size
    minutes = (words / words_per_minute).ceil
    "#{minutes} #{'minute'.pluralize(minutes)} read"
  end
end

Detailed Breakdown:

  • Automatic Availability: Helpers are automatically available in their corresponding views
  • Organization:
    • ApplicationHelper: Available to all views
    • PostsHelper: Only available in post views
  • Best Practices:
    • Keep helpers focused on presentation logic
    • Never put database queries in helpers
    • Name helpers clearly (e.g., format_date not date_thing)

üö® Common Mistakes:

  • Putting business logic in helpers
  • Creating huge helper files
  • Not using view components when appropriate

15. Background Jobs

‚è≥ When to Use Background Jobs?

  • Email sending
  • Image processing
  • Data exports
  • Third-party API calls

# app/jobs/image_processing_job.rb
class ImageProcessingJob < ApplicationJob
  queue_as :high_priority

  rescue_from(Error::Timeout) do
    retry_job wait: 5.minutes
  end

  def perform(user_id, image_path)
    user = User.find(user_id)
    ImageProcessor.new(user, image_path).process
  end
end

Job Lifecycle Explained:

  1. Enqueue: Add job to queue 
    
ImageProcessingJob.perform_later(current_user.id, uploaded_image.path)
  2. Processing: Job runner (Sidekiq/Resque) picks up job
  3. Execution: Runs perform method
  4. Completion/Retry: Marks job as done or retries on failure

‚ö†Ô∏è Job Safety Tips:

  • Jobs should be idempotent (safe to run multiple times)
  • Pass database IDs (not entire objects) to jobs
  • Set reasonable retry limits
  • Use separate queues for different priority jobs

16. View Components

üß© Component Benefits:

  • Reusable UI elements
  • Testable in isolation
  • Clear parameter interface
  • Better performance than partials

# app/components/alert_component.rb
class AlertComponent < ViewComponent::Base
  ERB_TEMPLATE = <<~ERB
    <div class="alert alert--<%= @type %>">
      <%= @message %>
    </div>
  ERB

  def initialize(type: :info, message:)
    @type = type
    @message = message
  end

  def call
    render inline: ERB_TEMPLATE, type: @type, message: @message
  end
end

Component Structure:

  • Initialization: Define clear input parameters
  • Templates: Can use separate ERB files or inline templates
  • Rendering: Explicitly define what to render in call
  • Testing: Test components with RSpec like regular classes

üîç When to Use Components:

  • Reused UI elements (buttons, cards)
  • Complex view logic
  • Need for view testing
  • Team collaboration (clear interfaces)
